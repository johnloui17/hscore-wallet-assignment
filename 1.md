# 🔐 Login Page — Implementation Plan
### User ID-based session, no passwords, no auth middleware

---

## The Flow (What We're Building)

```
User visits app
      │
      ▼
Is userId in localStorage?
  NO  → /login page (enter User ID)
  YES → /home (their wallets)
      │
      ▼
User types their ID (e.g. "john") → Enter
      │
      ▼
userId saved to localStorage → redirect to /
      │
      ▼
Home page: shows only THEIR wallets
      │
      ▼
Settings → Sign Out → clear localStorage → back to /login
```

---

## 🔀 Two Approaches — Pick One

### Option A — Frontend-Only (No Backend Changes) ⚡ Fast

Store a `userId → walletIds[]` map in localStorage.

```
localStorage = {
  "pocketfeel_user_id": "john",
  "pocketfeel_john_wallets": ["uuid1", "uuid2", "uuid3"]
}
```

- Login: save userId to localStorage
- Create wallet: add walletId to `pocketfeel_{userId}_wallets` array
- Home page: fetch only the wallet IDs in the user's list
- Works today, zero backend changes

**⚠️ Downside:** Wallets are browser-bound. Open the app on another device → empty list. If localStorage is cleared → wallets "lost" (data still in DB, just no pointer to it). Not ideal but acceptable for an assignment.

---

### Option B — Backend user_id Column (Proper Approach) ✅ Recommended

Add a `user_id` column to the `wallets` table. All API calls pass the userId.

```
GET  /api/v1/wallet?userId=john      → returns only john's wallets
POST /api/v1/wallet  { userId: "john", name: "..." }  → creates wallet for john
```

- Login: save userId to localStorage
- Every API call includes userId as a query param or in the body
- Home page: fetches `GET /wallet?userId=john` → only their wallets
- Works across devices, across browsers — data is truly per-user

**Backend changes needed:**
- Add `user_id` VARCHAR column to `wallets` (nullable for existing wallets)
- Update `getAllWallets(userId?)` service method to filter by userId
- Update `createWallet(userId?)` to save userId
- Update controller to accept `userId` query param

---

## 🏆 Recommendation

**Option B.** It's ~30 min of work, it properly separates data per user in the DB, and it's what a full-stack developer would actually build. The backend change is minimal.

```
│  └───────────────────────────┘  │
│                                 │
│  New here? Just pick any ID.   │

│                                 │
└─────────────────────────────────┘
```

- Input: plain text, any string (e.g. "john", "alice123")
- On submit: `localStorage.setItem('pocketfeel_user_id', userId)` → `router.push('/')`
- If input is empty → show inline error, don't proceed
- Existing user enters same ID → sees their wallets (stored in DB with that userId)
- New user enters new ID → sees empty wallet list → prompted to create first wallet

## 🔒 Route Protection (Middleware)

Next.js middleware runs before every page load:

```typescript
// src/middleware.ts
export function middleware(request: NextRequest) {
  const userId = request.cookies.get('pocketfeel_user_id')?.value;
  const isLoginPage = request.nextUrl.pathname === '/login';

  if (!userId && !isLoginPage) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
  if (userId && isLoginPage) {
    return NextResponse.redirect(new URL('/', request.url));
  }
}

export const config = {
  matcher: ['/', '/wallet/:path*', '/activity', '/cards', '/settings'],
};
```

> **Note:** Middleware reads cookies, not localStorage (localStorage is browser-only). So on login we also set a cookie alongside localStorage: `document.cookie = 'pocketfeel_user_id=john; path=/'`

## 🗄️ Backend Schema Change

```sql
ALTER TABLE wallets ADD COLUMN user_id VARCHAR(100);
CREATE INDEX idx_wallets_user_id ON wallets(user_id);
```

Existing wallets get `user_id = NULL` — they'll only be visible if you query without a userId filter (legacy behaviour stays intact).
